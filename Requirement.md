1. 프로젝트 개요

## 1. 프로젝트 개요

### 1-1. 프로젝트명

멀티자산 OMS 및 자동주문 시뮬레이션 플랫폼

### 1-2. 프로젝트 목적

이 프로젝트는 금융투자 산업의 실제 업무 흐름을 축소 구현해보며, 다음 세 가지를 직접 이해하고 체득하는 것을 목표로 한다.

- 산업 프로세스 이해
  포트폴리오 의사결정, 주문 생성, 사전 검증, 주문 처리, 체결, 사후 반영의 흐름을 이해한다.
- 요구사항 도출 역량 강화
  금융 시스템에서 왜 정합성, 통제, 상태 관리, 복구 가능성이 중요한지 직접 구현하며 파악한다.
- 개선 역량 강화
  단순 기능 구현이 아니라, 실제 시스템 운영을 가정했을 때 어떤 구조가 더 확장 가능하고 안전한지 판단하는 역량을 쌓는다.

### 1-3. 프로젝트 핵심 관점

이 프로젝트의 핵심은 수익률을 높이는 자동매매 전략 개발이 아니다. 핵심은 다음과 같다.

- 여러 주문 발생 원천을 하나의 파이프라인으로 처리하는 구조
- 주문 상태 전이의 일관성
- 사전 리스크 통제
- 체결 후 포지션/현금/손익 반영의 정합성
- 이벤트 기록 기반 추적과 재현 가능성

## 2. 해결하고자 하는 문제

금융 시스템에서 주문은 단순 요청이 아니다. 하나의 주문은 다음과 같은 문제를 동반한다.

- 누가 왜 이 주문을 만들었는가
- 이 주문이 규칙 위반은 아닌가
- 외부 시장으로 나간 후 현재 상태는 무엇인가
- 부분체결, 취소, 거절이 발생했을 때 어떻게 반영할 것인가
- 현재 포지션과 손익은 정확한가
- 장애 발생 후 과거 이벤트를 재현할 수 있는가

즉, 금융 시스템은 "주문 넣고 끝"이 아니라 의도 생성부터 사후 반영까지 이어지는 긴 상태 관리 문제다.

이 프로젝트는 이 문제를 직접 구현해보면서, 업무 프로세스가 시스템 구조를 어떻게 결정하는지 보여주는 것을 목표로 한다.

## 3. 산업 프로세스와 시스템 대응

여기서는 실제 업무 흐름이 시스템에서 어떻게 번역되는지 정리한다.

### 용어(초안)

- Order Intent (`OrderIntent`)
  리밸런싱/전략/수동 입력 등 다양한 소스가 공통으로 생성하는 "주문 의도" 표현. 이 단계는 아직 실제 거래소/브로커 주문이 아니다.
- Pre-trade Risk / Compliance Layer
  주문이 한도/규정/중복/가격 밴드 등을 위반하는지 사전에 검증하는 계층.
- OMS Core
  승인된 주문의 생성, 상태 전이, 수정/취소 요청, 이벤트 저장 등을 일관되게 담당하는 코어.
- Execution Simulator / Venue Adapter
  외부 시장(거래소/브로커)의 응답을 시뮬레이션하거나 어댑터로 추상화하는 계층.
- Post-trade Processor
  체결 결과를 바탕으로 포지션/현금/손익/이력을 정합하게 반영하는 계층.

### 3-1. 투자 의사결정

업무에서는 포트폴리오 목표 비중, 전략 신호, 트레이더 판단 등을 바탕으로 "무엇을 얼마나 매수/매도할지"가 결정된다.

시스템에서는 이 단계를 Order Intent 생성으로 본다. 즉, 다음과 같은 주체가 `OrderIntent`를 만들 수 있다.

- 리밸런싱 엔진
- 자동매매 전략
- 사용자의 수동 입력

하지만 이 단계에서는 아직 실제 거래소 주문이 아니다.

### 3-2. 사전 검증

업무에서는 주문이 규정 위반인지, 한도 초과인지, 중복인지 등을 확인한다.

시스템에서는 이 단계를 Pre-trade Risk / Compliance Layer로 분리한다. 예시 검증 항목은 다음과 같다.

- 주문 수량 한도 체크
- 총 포지션 한도 체크
- 중복 오픈 주문 체크
- 가격 밴드 체크
- kill switch 체크

등을 독립된 규칙으로 평가한다.

### 3-3. 주문 관리

업무에서는 승인된 주문이 실제로 전송되고, 정정·취소·상태확인이 이뤄진다.

시스템에서는 이 단계를 OMS Core가 담당한다. OMS Core의 책임은 다음과 같다.

- 주문 생성
- 주문 상태 전이 관리
- 주문 수정/취소 요청 처리
- 중복 요청 방지
- 주문 이벤트 저장

### 3-4. 체결 처리

업무에서는 외부 시장이 체결 여부를 결정한다. 즉, 내 시스템이 주문을 보냈다고 바로 체결이 되는 게 아니다.

시스템에서는 이 단계를 Execution Simulator / Venue Adapter로 표현한다. 여기서 발생 가능한 이벤트는 다음과 같다.

- `ACK`
- `PARTIAL_FILL`
- `FILL`
- `REJECT`
- `CANCEL_SUCCESS`
- `CANCEL_FAIL`

### 3-5. 사후 반영

업무에서는 체결 결과를 바탕으로 포지션, 현금, 손익, 이력을 반영한다.

시스템에서는 이 단계를 Post-trade Processor로 분리한다. 즉, 아래 항목을 여기서 수행한다.

- 보유 수량 변경
- 평균단가 갱신
- 현금 반영
- 실현손익/평가손익 계산
- 감사 로그 저장

## 4. 사용자와 주요 시나리오

### 4-1. 가정하는 사용자

이 프로젝트는 실제 서비스 사용자보다 금융 업무 역할을 단순화한 "가상 사용자"를 상정한다.

| 사용자(역할) | 주요 행위 |
| --- | --- |
| Portfolio Manager | 목표 비중 설정, 리밸런싱 실행 |
| Strategy Engine | 시장 데이터 기반 신호 생성, 자동주문 의도 생성 |
| Trader / Operator | 수동 주문 입력, 주문 상태 확인, 취소 요청, 장애 상황 점검 |
| Risk Manager | 리스크 규칙 설정, 위반 주문 확인 |

### 4-2. 핵심 업무 시나리오

#### 시나리오 A. 리밸런싱 주문

- 사용자가 목표 비중을 설정한다.
- 시스템이 현재 포지션과 시세를 기준으로 부족/초과 수량을 계산한다.
- Order Intent를 생성한다.
- 리스크 검사를 수행한다.
- 승인되면 주문이 생성된다.
- 체결 결과가 도착한다.
- 포지션과 손익이 반영된다.

#### 시나리오 B. 자동매매 주문

- 전략 엔진이 시장 데이터 리플레이를 수신한다.
- 전략 조건 충족 시 주문 의도를 생성한다.
- 동일 종목 오픈 주문 여부를 확인한다.
- 승인되면 주문이 전송된다.
- 부분체결/전량체결이 발생한다.
- 포지션과 현금이 반영된다.

#### 시나리오 C. 수동 취소

- 운영자가 미체결 주문을 조회한다.
- 취소 요청을 보낸다.
- 주문 상태가 `CANCEL_REQUESTED`로 바뀐다.
- 시장 응답에 따라 `CANCELED` 또는 추가 체결이 발생한다.
- 최종 상태를 확정한다.

#### 시나리오 D. 장애/재현

- 특정 주문에서 상태 불일치가 발견된다.
- 시스템이 주문 이벤트 로그를 조회한다.
- 이벤트를 시간순으로 재생한다.
- 어떤 이벤트가 현재 상태를 만들었는지 재현한다.

## 5. 프로젝트 구조 요구사항

프로젝트는 다음 계층으로 분리되어야 한다.

### 5-1. Market Data Layer

역할은 다음과 같다.

- 종목 정보 관리
- 가격 정보 관리
- 리플레이 데이터 공급

### 5-2. Intent Generation Layer

역할은 다음과 같다.

- 수동 주문 생성
- 리밸런싱 주문 생성
- 전략 기반 주문 생성

### 5-3. Pre-trade Risk Layer

역할은 다음과 같다.

- 규칙 검증
- 승인/거절 판정
- 실패 사유 저장

### 5-4. OMS Core

역할은 다음과 같다.

- 주문 생성
- 상태 관리
- 이벤트 저장
- 취소/정정 요청 처리

### 5-5. Execution Layer

역할은 다음과 같다.

- 외부 시장 역할 대체
- 체결 이벤트 생성

### 5-6. Post-trade Layer

역할은 다음과 같다.

- 포지션/현금/손익 반영

### 5-7. Audit / Replay Layer

역할은 다음과 같다.

- 이력 조회
- 상태 재현
- 장애 분석

### 5-8. 프로젝트 폴더 구성

```text
├─ market-data
├─ intent-generation
│  ├─ manual-input
│  ├─ rebalancing
│  └─ strategy
├─ pre-trade-risk
├─ oms-core
│  ├─ domain
│  ├─ application
│  ├─ infra
│  └─ api
├─ execution
├─ post-trade
├─ audit-replay
└─ admin-dashboard
```

`admin-dashboard`는 핵심 처리 계층은 아니지만, Trader / Operator가 수동 주문 입력, 주문 상태 확인, 취소 요청, 장애 상황 점검을 수행하는 운영자 인터페이스로 둔다.
